## 0. 설계 원칙 (코딩 전체에 걸쳐 반드시 준수)

### 0.1 OpenMV / PC Math Parity 원칙

> **이름이 같은 함수·클래스는 로직이 동일해야 한다.**

- 주(主) 연산 장치는 **OpenMV (MicroPython)**. MicroPython의 API 제약이 알고리즘의 기준선이다.
- PC(VVM)에서 동일 이름의 함수를 구현할 때, `cv2`, `scipy` 등 OpenMV에 없는 고급 API로 대체하거나 확장하면 **안 된다**.
- PC 쪽에서만 사용하는 신규 알고리즘이 필요할 때는 **새 이름**을 부여한다. `HyLine_PC` 같은 `~_PC` 접미사 패턴은 **금지**.
- 같은 이름을 공유하는 모든 툴은 `usable_devices` 딕셔너리로 장치별 실행 가능 여부를 런타임에 검증받는다.

### 0.2 장치별 `usable_devices` 플래그

모든 `HyTool` 서브클래스는 클래스 변수 또는 인스턴스 속성으로 `usable_devices`를 보유한다.

```python
class HyLine(HyTool):
    usable_devices = {'OpenMV': True, 'PC': True, 'VVM': True}

class HySomeNewPCOnlyTool(HyTool):
    usable_devices = {'OpenMV': False, 'PC': True, 'VVM': True}
```

- `exec_dev_id`가 할당된 장치에서 `usable_devices[my_device_type]`이 `False`이면 해당 노드는 `EXEC_ERROR` 상태로 전환되고, 오류 로그를 남긴다.
- 신규 툴 추가 시 `usable_devices` 선언은 **필수**. 누락 시 `True` 기본값으로 간주하지 않는다(런타임 경고).

### 0.3 구현 중 계획 변경 허용 범위

> **Phase 1 이하의 세부 항목(§3~§6)은 코딩 진행 중에 언제든 수정될 수 있다.**

- 인터페이스·시그니처·속성명 등은 구현 중 발견되는 제약에 따라 조정 가능.
- 단, §2 이름 마이그레이션 결과와 §0 설계 원칙은 **불변**.
- WBS 수정 시 해당 항목에 `[수정됨]` 태그를 붙이고 이유를 1줄 추가한다.
