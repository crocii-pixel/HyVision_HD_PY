# 고급 로직 및 측정 툴 상세 명세서 (Advanced Logic & Measurement Tools)

본 문서는 HyVision Pro-Inspector 시스템의 핵심 경쟁력인 **동적 앵커 추종 툴(`HyLocator`)**, **조건부 지연 타이머 툴(`HyWhen`)**, **논리 컨테이너 툴(`HyAnd`, `HyOr`)**, 그리고 **직관적 가상 블록 기반 디스플레이 판독 툴(`HyFND`)**의 상세한 개념, 작동 원리, 응용 시나리오를 명세합니다.
HyLine, HyPatMat등 기 작성된 기능은 legacy 코드의 로직과 기능의 맥락을 그대로 사용하여, 기능적 동일성을 확보합니다.

## 1. HyLocator (마스터 앵커 추종 및 고정 정책)

`HyLocator`는 화면 전체의 공간 좌표계를 쥐고 있는 마스터 앵커입니다. 실제 검사 현장에서 부품이 정지하더라도 발생하는 미세한 진동이나 흔들림에 대응하기 위해, 1회 검사 사이클(Cycle) 동안 앵커의 좌표를 어떻게 취급할지 결정하는 강력한 정책을 제공합니다.

### 1.1. 사이클 내 앵커 갱신 모드 (Update Policy)

- **[모드 A] Continuous Update (지속 추종):** * 사이클이 진행되는 동안, 매 프레임마다 `HyLocator`가 새로운 앵커 좌표를 실시간으로 다시 찾아서 갱신합니다.
  - **목적:** 부품이 멈추지 않고 미세하게 움직이거나 진동하고 있을 때, 종속된 측정 툴(Line, FND 등)의 ROI 박스도 그 움직임을 실시간으로 따라가며(Tracking) 검사를 수행하게 합니다.
- **[모드 B] Cycle Lock (1회 고정):**
  - 부품이 진입하여 사이클이 시작될 때 단 한 번만 앵커 좌표를 획득하고, 이번 사이클이 완전히 끝날 때까지 그 좌표를 꽉 물고 고정(Lock)합니다.
  - **목적:** 부품이 완벽히 정위치에 고정되었음이 보장되거나, 오히려 진동에 의해 앵커가 흔들려서 후속 정밀 연산이 틀어지는 것을 방지하고 싶을 때 사용합니다. 연산량도 대폭 절약됩니다.

## 2. HyWhen (조건부 지연 타이머 및 트리거)

`HyWhen`은 단순한 분기문이 아니라, PLC의 **TON (Timer On-Delay)** 개념을 소프트웨어로 구현한 **'조건부 지연 타이머 및 상태 감시자'**입니다. 기계적 움직임이 안정화될 때까지의 '타이밍'을 통제하고, 자신의 결과값을 능동적으로 선택합니다.

### 2.1. 핵심 파라미터 (Core Parameters)

- **Target Tool ID:** 감시할 대상 툴의 ID.
- **Condition:** 타겟 툴이 도달해야 할 상태 (예: `rst_state == True` (인식됨) 또는 `False` (사라짐/Invert)).
- **Timeout (지연 시간):** 타겟 툴이 조건을 만족한 **'그 기점'으로부터, 하위 툴을 실행하기 전까지 의도적으로 기다려주는 지연 시간(ms) 또는 프레임 수**.
- **💡 Output Selection (출력 결정 모드):** `HyWhen`이 실행을 마친 후, 상위 트리로 보고할 자신의 최종 결과(`rst_state`)를 어떻게 결정할지 선택합니다.
    1. **내부 툴 처리 여부 (Execution Status):** 하위 툴이 에러 없이 무사히 '실행되었는가(`rst_done`)' 자체를 True/False로 반환합니다.
    2. **내부 툴 논리 결과 (Logical Result):** 하위 툴이 최종적으로 판정한 '합/불 결과(`rst_state`)'를 그대로 끌어와 자신의 결과로 반환합니다.

### 2.2. 작동 원리 (Pending 상태의 중요성)

1. **감시 및 미처리 대기 (Pending):** `HyWhen`은 타겟 툴이 조건을 만족할 때까지 감시합니다. 이 기간 동안 `HyWhen`은 실패(NG)가 아니라 **'미처리/진행 중(Pending)'** 상태를 유지합니다.
2. **조건 충족 및 타이머 시작:** 타겟 툴이 조건에 부합하면, 즉시 하위 툴을 실행하지 않고 내부 타이머(Timeout)를 가동합니다. (이때도 상태는 여전히 Pending입니다.)
3. **지연 대기 (Delay):** Timeout으로 설정된 시간 동안 시스템은 대기합니다.
4. **실행 및 결과 도출 (Triggered):** 지연 시간이 경과하면 하위 노드를 실행시키고, 설정된 `Output Selection`에 따라 최종 결과를 확정 지어 보고합니다.

## 3. 논리 컨테이너 (HyAnd, HyOr)와 분기 처리 (Branching)

`HyWhen`의 유연한 상태(Pending) 보고 시스템은, 상위 로직 컨테이너인 `HyAnd` 및 `HyOr`와 결합할 때 진정한 위력을 발휘합니다.

### 3.1. HyAnd의 상태 지연 전파 (Pending Propagation)

- **상황:** `HyAnd` 컨테이너 내부에 `HyWhen`이 자식 노드로 존재합니다.
- **작동 논리:** * 일반적인 `HyAnd`는 자식 중 하나라도 False가 나오면 즉시 전체를 False로 단축 평가(Short-circuit)합니다.
  - 하지만 자식인 `HyWhen`이 아직 타겟 조건을 충족하지 못해 **'미처리/진행 중(Pending)'** 상태를 보고하고 있다면, **`HyAnd` 역시 섣불리 단축 평가를 내리거나 실패로 단정 짓지 않습니다.**
  - `HyAnd`는 전체 시퀀스를 **'진행 중'** 상태로 유지하며, `HyWhen`이 최종 결정을 내릴 때까지 묵묵히 동기화되어 기다립니다. 이를 통해 런타임 중 데이터가 확정되지 않아 발생하는 섣부른 오판을 완벽하게 방어합니다.

### 3.2. 혼류 라인 스마트 라우팅 (HyOr 분기 처리)

- `HyOr` 내부에 여러 개의 `HyWhen`을 배치하면, 프로그래밍의 `switch-case`나 `if-elif`와 같은 **조건부 분기(Branching)** 처리를 시각적인 노드 트리로 완벽하게 구현할 수 있습니다.
- **[트리 구성]** `HyOr_Master`
  - ┣ `HyWhen_A` (Target: A모델 마크) ➔ 하위 툴: A모델 전용 치수 검사
  - ┗ `HyWhen_B` (Target: B모델 마크) ➔ 하위 툴: B모델 전용 치수 검사

### 3.3. 💡 복합 응용 시나리오: 동적 기기 시퀀스 및 버전 검증 (상태 머신 구현)

개발자님이 제시한 이 시나리오는 `HyAnd`(순서 강제), `HyWhen`(Timeout 대기), `HyFND`(값 검증)를 결합하여 **디스플레이가 변하는 기기의 "진행 순서", "처리 속도", "버전/상태값"을 동시에 검증**하는 하이엔드 로직입니다.

- **[시나리오]** 기기의 부팅/작동 과정을 실시간으로 지켜보며 3단계(ver ➔ aaa ➔ bbb)를 순차적으로 검증합니다.
- **[로직 트리 구성]**

    ```
    HyAnd_Master (순차 진행 강제. 하나라도 실패/시간초과 시 전체 NG)
      ┣ HyWhen_1 (Target: HyFND_A == 'ver', Timeout: 2000ms, Output: 자식 결과)
      ┃  ┗ HyFND_B (모드: Equal, 기준값: 'v101')
      ┣ HyWhen_2 (Target: HyFND_A == 'aaa', Timeout: 3000ms, Output: 자식 결과)
      ┃  ┗ HyFND_B (모드: Min/Max, 기준값: 10 ~ 100)
      ┗ HyWhen_3 (Target: HyFND_A == 'bbb', Timeout: 3000ms, Output: 자식 결과)
         ┗ HyFND_B (모드: Min/Max, 기준값: 1000 ~ 2000)
    ```

- **[불량 판정(NG) 방어 메커니즘]**
    1. **값/버전 불량 (Value Error):** `HyFND_A`에 정상적으로 'ver'이 떴으나, `HyFND_B`를 읽어보니 'v102'인 경우 ➔ `HyFND_B` 실패 ➔ 전체 NG.
    2. **진행 순서 불량 (Sequence Error):** 'ver' 화면 이후 'aaa' 단계를 건너뛰고 바로 'bbb' 화면으로 넘어간 경우 ➔ `HyAnd`는 순차 평가를 하므로 `HyWhen_2`가 'aaa'를 계속 기다림 ➔ 'aaa'가 안 나오므로 **Timeout 발생** ➔ 전체 NG.
    3. **처리 속도 지연 (Timeout Error):** 기기의 처리 속도가 느려져 지정된 3000ms 이내에 'aaa' 화면으로 전환되지 않는 경우 ➔ `HyWhen_2`의 **Timeout 발생** ➔ 전체 NG.

이처럼 단순한 이미지 비교를 넘어, 시간의 흐름에 따른 **절차적 무결성(Procedural Integrity)**까지 검증해 내는 완벽한 상태 머신(State Machine)을 소프트웨어적으로 구축할 수 있습니다.

## 4. HyFND (가상 블록 기반 7-세그먼트 판독기)

`HyFND`는 디지털 계기판(7-Segment, 14-Segment 등)을 판독하는 툴입니다. 머신러닝/OCR을 쓰지 않고도, 사용자가 직관적인 UI를 통해 가상 블록을 그려 빠르고 정확하게 숫자를 판독하도록 설계되었습니다.

### 4.1. UI 설정 및 구성 시나리오 (UI/UX Workflow)

작업자가 매뉴얼 없이도 쉽게 세팅할 수 있도록, 단계별 오버레이(Overlay) UI를 제공합니다.

- **[Step 1] 마스터 ROI 영역 설정:** 계기판 전체 영역에 커다란 사각 박스(Master ROI)를 칩니다.
- **[Step 2] 자릿수(Digits) 입력:** "읽어낼 글자의 개수(예: 4자리)"를 입력하면, 마스터 ROI 내부가 4개의 균등한 하위 박스(Digit Box)로 자동 분할됩니다.
- **[Step 3] 가상 세그먼트 블록(Grid) 오버레이:** 각각의 Digit Box 내부에 '8'자 모양을 구성하는 7개의 세그먼트 선(A~G)과 소수점(DP) 위치에 얇은 가상 블록(Virtual Block)들이 오버레이로 나타납니다.
- **[Step 4] 블록 미세 조정 (Fine-Tuning):** 실제 계기판의 글꼴이나 기울기에 맞춰, 사용자가 이 가상 블록들의 **두께(Thickness), 간격, 그리고 전체적인 기울기(Skew/Italic)**를 조절하여 LED 세그먼트 위에 정확히 겹치도록 맞춥니다.

### 4.2. 판독 로직 (Decoding Logic)

1. **블록별 밝기 평가:** 각 가상 블록(A~G) 영역 내부 픽셀들의 평균 밝기를 구해, 사용자가 설정한 임계값(Threshold)을 넘으면 해당 세그먼트가 'ON(1)'된 것으로 판정합니다.
2. **비트맵 매칭 (Bitmap Matching):** 7개의 ON/OFF 상태를 7비트 이진수로 변환하여 미리 정의된 룩업 테이블(LUT)과 대조, 숫자를 디코딩합니다.
3. **최종 판정 (Judgement):**
    - **Equal 모드:** 판독된 전체 숫자가 타겟 문자열(예: "220")과 일치하는지 판정합니다.
    - **Min/Max 모드:** 판독된 결과를 실수(Float)로 변환하여 상/하한치(예: 215.0 ~ 225.0) 범위 내에 들어오는지 판정합니다.
