# HyVision Pro-Inspector 통합 시스템 마스터 명세서

**버전:** 4.3 (DCOM 트리 구조 동기화 및 상태(State) 동기화 메커니즘 반영)
**개요:** 본 문서는 마이크로컨트롤러 기반 비전 센서(Edge)와 Host PC를 결합한 하이브리드 비전 검사 시스템의 철학, 아키텍처, 통신 프로토콜, 클래스 설계, 4대 구동 모드, UI/렌더링 로직, 그리고 테스트 계획을 하나로 통합한 최종 명세서입니다.

## 1. 핵심 설계 철학 (Core Philosophy)

### 1.1. DCOM 기반 단일 논리 집행관 트리 (Distributed Single Logical Executor)

본 시스템은 기존의 수직적인 'Host-Slave' 구조를 탈피합니다. 전체 비전 네트워크(PC, 다수의 카메라 노드)가 **물리적으로 분리되어 있지만, 논리적으로는 완벽히 동일한 하나의 '집행관 트리(Executor Tree)'를 공유**합니다.

### 1.2. 바톤 터치와 파이프라인 병렬 처리 (Pipelined Parallel Processing)

- **바톤 터치:** 트리를 순회하는 도중 툴의 `device_id`가 자신의 ID와 일치하면 연산을 수행하고, 일치하지 않으면 현재까지의 상태(State)와 이미지 컨텍스트를 다음 장치로 넘깁니다.
- **병렬 파이프라인:** 엣지 카메라가 PC로 바톤과 이미지를 넘겨준 직후 대기하지 않습니다. 즉시 **다음 대상물(Cycle N+1)의 영상 캡처 및 앞단(Front-end) 물리 연산을 병렬로 시작**하여 시스템의 전체 쓰루풋(UPH)을 극대화합니다.

### 1.3. 단일 마스터 앵커와 직관적 픽스쳐 (Single Anchor Fixture)

사용자 경험(UX) 극대화를 위해, 공간 좌표계의 물리적 변환 기준을 제공하는 마스터 `HyLocator`는 시스템에 단 1개만 존재합니다. 나머지 모든 툴은 `use_anchor` (True/False) 속성 하나만으로 종속 여부를 결정합니다.

### 1.4. 시스템 인덱싱 순환 및 저장소 원칙

- **1-Based Indexing & ID Cycling:** 시스템 내의 모든 식별자(Tool ID, Seq ID 등)는 1부터 시작합니다. 특히 `tx_id`, `cycle_id`, `img_id` 등은 메모리 최적화를 위해 무한히 커지지 않고 **순환(Cycling)**하여 중복만 회피합니다.
- **이미지 소스 중심 저장소:** 레시피는 영상을 생산하는 소스 장치(물리 카메라는 Flash 메모리, 에뮬레이터는 PC 로컬 디스크)에 저장되어 독립 구동을 보장합니다.

## 2. 통합 연결 아키텍처 및 가상 캠 (Virtual Cam Routing)

모든 연결을 **단일 연결 리스트 (현 프로젝트에서는 COM 포트 리스트)**로 통합하여 극강의 직관성을 제공합니다.

### 2.1. 가상 캠 생성 및 통합 리스트

- **가상 캠(Virtual Cam) 생성:** 사용자는 UI를 통해 새로운 '가상 캠'을 생성하고, 이를 특정 '로컬 폴더(이미지 셋)'나 'PC 연결 웹캠'에 바인딩할 수 있습니다.
- 사용자가 생성한 '가상 캠(Virtual Cam)'을 물리적 COM 포트와 함께 **단일 연결 드롭다운 리스트**로 통합 제공하여 극강의 직관성을 보장합니다.
- **동등한 시민권 (Equal Citizenship):** 이렇게 생성된 가상 캠들은 물리적 OpenMV 장비의 USB COM 포트들과 함께 섞여 **단일 연결 드롭다운 리스트(Port List)**에 나열됩니다.
- **투명성:** 가상 캠은 스스로를 완벽한 독립 비전 머신(`Device 1`)으로 인식하며, UI와 시스템은 선택된 포트가 물리 장치인지 가상 장치인지 알 필요 없이 동일한 프로토콜로 통신합니다.

### 2.2. 이원화된 비전 엔진 (VVM vs UI Preview Engine)

- **VVM (Virtual Vision Machine):** 가상 캠이 선택되었을 때, 통신망(Queue) 뒤편에서 물리 장치를 100% 동일하게 흉내 내는 백그라운드 스레드 엔진입니다.
- **UI Preview Engine:** 이와 별개로, TEACH 모드에서 사용자가 툴 파라미터를 조작할 때 통신 지연 없이 즉각적인 화면 렌더링(미리보기)을 제공하기 위해 UI 스레드 측에 탑재되는 1:1 복제 비전 엔진입니다. 두 엔진은 역할이 완벽히 분리됩니다.

## 3. 통합 제어 및 DCOM 트리 동기화 프로토콜 (Tree Sync Protocol)

시스템의 통신은 단순한 명령 전달을 넘어, 1) 트리의 구조(DNA)를 복제하고, 2) 실행 중인 트리의 상태(State)를 양방향으로 동기화하는 완벽한 DCOM(분산 객체) 프레임워크입니다.

### 3.1. [트리 구조 생성/CRUD] 32바이트 Command Struct

PC가 작성한 레시피(트리)를 장비로 전송할 때, 최상위 노드부터 DFS 방식으로 순차 송출하여 장비가 트리를 재조립할 수 있도록 **`Parent ID`** 정보를 패킷에 숨겨 보냅니다 (Tree Serialization).

- **포맷:** `<H B B 4i I 8x` (리틀 엔디안)

| Offset | 데이터형 | 변수명 | 설명 |
| --- | --- | --- | --- |
| `0~1` | `uint16` | **Sync Header** | 고정값 `0xBB66` |
| `2` | `uint8` | **Cmd ID** | `0x10` (SET_TOOL) 등 제어 명령 |
| `3` | `uint8` | **Target ID** | 툴(노드) 고유 ID (`tool_id`) |
| `4~7` | `int32` | **Param 1** | 💡 **[트리 조립 정보] 상위 16bit: `device_id`, 하위 16bit: `parent_tool_id` (0이면 Root 노드)** |
| `8~19` | `int32 * 3` | **Param 2~4** | `tool_type` 및 로컬 파라미터 (좌표, 임계값 등 압축) |
| `20~23` | `uint32` | **Payload Len** | 템플릿 등 추가 바이너리 크기 |
| `24~31` | `Pad` | **Padding** | 8바이트 예약 여백 |

### 3.2. [트리 실행 상태 동기화] 64바이트 Result Struct (Burst)

장비가 자신의 턴을 마치고 PC로 바톤을 넘길 때, 자신이 실행했거나 스킵(Short-circuit)한 **모든 노드의 상태를 64바이트 패킷 배열(Burst)로 송출**합니다. PC는 이 패킷들을 받아 자신의 로컬 트리에 1:1로 덮어쓰기(State Injection)하여 완벽한 상태 동기화를 이룹니다.

- 전송 형태: `[Sync 0xAA55(2B)]` + `[패킷 N(2B)]` + `[64B 패킷 N개]` + `[이미지 크기 M(4B)]` + `[M바이트 이미지]`
- **64바이트 단일 패킷 포맷:** `<HIIIHHHBB5f4fI2x`

| Offset | 데이터형 | 변수명 | 설명 |
| --- | --- | --- | --- |
| `2~5` | `uint32` | **tx_id** | 1~1000 순환 번호 |
| `6~9` | `uint32` | **cycle_id** | 검사 대상품 고유 번호 (병렬 처리 컨텍스트 식별자) |
| `14~15` | `uint16` | **seq_id** | 트리 내 실행 순서 |
| `16~17` | `uint16` | **tool_id** | 💡 **[상태 매핑 키] 이 ID를 가진 PC 측 트리의 노드를 찾아 상태 갱신** |
| `18~19` | `uint16` | **exec_dev_id** | 연산 할당 디바이스 ID (1:Cam, 2:PC) |
| `20` | `uint8` | **rst_done** | 💡 **[실행 상태] 0: 미실행/단축스킵, 1: 정상 완료, 2: 에러 발생** |
| `21` | `uint8` | **rst_state** | 💡 **[논리 상태] 노드의 논리 판정 결과 (0: NG/False, 1: OK/True)** |
| `22~41` | `float * 5` | **Geometry** | `x, y, w, h, angle` (결과 좌표) |
| `42~57` | `float * 4` | **Stats** | `stat1~4` (Score 등) |

## 4. 4대 구동 모드 상세 시나리오 (4 Operating Modes)

장비의 제어와 UI의 표출 방식은 철저히 4가지 모드로 구분되어 작동합니다.

### 4.1. LIVE 모드 (영상 셋업 모드)

- **역할:** 카메라 렌즈 초점, 조명, 노출, 게인 등 하드웨어적/광학적 영상 조건을 세팅합니다.
- **동작:** 최고 화질의 영상이 실시간 송출됩니다. (비전 연산 OFF)
- **데이터 보존:** 이 모드에서 변경된 영상 설정(Exposure, Gain 등)은 사용자가 명시적으로 '저장(Save)'을 클릭하지 않으면, 모드 이탈 시 물리적으로 원상 복구(Rollback)됩니다.

### 4.2. TEACH 모드 (레시피 및 툴 편집 모드)

- **역할:** 실제 검사에 사용될 비전/논리 툴을 생성, 배치, 수정하고 시나리오 트리를 짭니다.
- **동작:** * 영상이 정지되며 **단 한 장의 이미지** 위에서 모든 작업이 이루어집니다. (필요 시 새로운 이미지를 1장씩 갱신 요청 가능)
  - 사용자가 툴(ROI, 임계값 등)을 조작하면, **UI Preview Engine**이 즉각적으로 연산하여 렌더링 결과를 미리보기(Preview)로 제공합니다.

### 4.3. TEST 모드 (출력 및 UI OSD 설정 모드)

- **역할:** 실제 실행(RUN) 시 결과를 어디로 배출할지, 화면을 어떻게 꾸밀지(OSD: On-Screen Display) 설정합니다.
- **설정 항목:**
    1. **HyFin 출력 매핑:** 각 `HyFin` 노드의 최종 판정 결과를 장치의 어떤 물리 핀(I/O Pin) 또는 통신 포트로 배출할지 매핑합니다.
    2. **영상 송출 주기:** 매 프레임 송출, 특정 주기(예: 1초에 1장), 혹은 영상 송출 끄기(결과만 수신)를 결정합니다.
    3. **최종 결과창(Status Box):** 첫 번째 `HyFin`의 결과(True/False)를 거대한 OK/NG 텍스트로 화면 어디에 띄울지 크기와 위치를 설정합니다.
    4. **상세 결과창(Spec Box):** 각 비전 툴의 세부 결과(치수, 점수 등)를 나열하는 텍스트 목록창의 위치를 설정합니다.
    5. **꾸밈 요소(Decorations):** 각 툴의 ROI 박스 선 색상, 십자선 마커 등 시각적 전략 요소의 표시 여부를 토글합니다.

### 4.4. RUN 모드 (실전 검사 모드)

- **역할:** 양산 라인에서 실제로 시스템이 가동되는 모드입니다.
- **동작:** * TEST 모드에서 설정한 모든 출력(I/O)과 UI OSD 규칙에 따라 화면이 렌더링됩니다.
  - 영상 송출을 껐을 경우, 카메라는 결과 패킷(Burst)만 초고속으로 전송하며, PC 화면에는 이전에 캡처된 배경 위에 **[최종 결과창]과 [상세 결과창] 데이터만 갱신되어 표시**됩니다.

## 5. 소프트웨어 클래스 및 비전 툴 라이브러리

### 5.1. 물리 비전 툴 (Physical Vision Tools) - 주 할당: Device (1)

- **`HyLocator`**: 화면 전체 좌표계 기준을 잡는 단일 마스터 앵커 툴.
- **`HyLine`**: 단일 선 추출 및 `midpoint_check` 기반 노이즈 방어.
- **`HyPatMat`**: NCC 기반 템플릿 영역 탐색.
- **`HyIntersection`**: 두 기하 객체의 교차점 도출.
- **`HyLinePatMat`**: `HyLine`으로 각도를 찾은 뒤, 역방향 크롭(Inverse Crop)을 수행하여 회전 왜곡 없이 템플릿 매칭.

### 5.2. 정밀 측정 툴 (Measurement Tools) - 주 할당: PC (2)

- **`HyDistance`**: 픽셀/물리 거리 및 평행도(Max Angle Diff) 측정.
- **`HyContrast`**: 밝기, 표준편차 분석을 통한 스크래치/도포 불량 판정.
- **`HyFND`**: 7-세그먼트 가상 블록 스캔 및 문자/숫자 매칭 판독.

### 5.3. 로직 집행관 툴 (Logic Executor Tools) - 주 할당: PC (2)

- **`HyWhen`**: 타겟 툴 결과가 참일 때만 하위 노드를 실행(단축 평가 트리거).
- **`HyAnd` / `HyOr`**: 논리 게이트. 조건 충족 시 즉시 하위 연산 스킵(Pruning).
- **`HyFin`**: 사이클 종료자. 최종 판정을 도출하고 TEST 모드에서 매핑된 I/O 핀 및 화면(Status Box)으로 결과를 브로드캐스트.

## 6. 렌더링 및 동적 좌표계(Fixture) 최적화

- **Top-Left 기준 아핀 변환:** 마스터 앵커의 편차($\Delta x, \Delta y, \Delta \theta$)를 종속 툴의 Top-Left 원점 기준 `QTransform` 행렬로 1회 연산하여 정밀 Fixture 렌더링.
- **역방향 크롭 (Inverse Crop Warping):** 원본 이미지의 탐색 영역만 역방향($-\Delta\theta$)으로 회전 크롭하여 왜곡(블랙 패딩) 없는 매칭 수행.
