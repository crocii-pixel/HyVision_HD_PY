# 프로토콜 명세서 (Protocol Specification)

**버전:** 3.0  
**최종 수정:** 2026-04-16  
**변경 이력:**
- v2.0 (2026-04-14): 마스터 상세서 v4.3 동기화, Tri-State 체계 도입
- v3.0 (2026-04-16): Command Struct 전면 개정 — `target_type` 헤더 승격, `fparam` 추가, 전 필드 2/4바이트 자연 정렬

## 1. 설계 원칙

### 1.1. 고정 길이 이진 구조체 (Fixed-Length Binary Struct)

가변 길이 문자열(JSON 등) 파싱으로 인한 마이크로컨트롤러의 메모리 파편화 및 병목 현상을 원천 차단하기 위해, 모든 런타임 통신에 **고정 길이 이진 구조체**를 사용합니다.

### 1.2. 바이트 오더 (Byte Order)

- **모든 구조체는 리틀 엔디안(Little-Endian, `<`)** 으로 인코딩/디코딩합니다.
- PC(x86/x64)와 ARM Cortex-M7(OpenMV H7 Plus) 모두 리틀 엔디안 네이티브이므로, 바이트 스왑 없이 직접 캐스팅 가능합니다.

### 1.3. 통신 방향 및 채널 크기

| 방향 | 채널명 | 단위 크기 | 설명 |
|------|--------|----------|------|
| **PC ➔ Device** | 제어 채널 (Command Channel) | `32 Bytes` 고정 헤더 + 선택적 바이너리 페이로드 | 모드 전환, 레시피(트리) 주입 |
| **Device ➔ PC** | 결과 채널 (Result Channel) | `64 Bytes` × N개 배열(Burst) + 영상 페이로드 | 트리 상태 동기화, 이미지 전송 |

## 2. [PC ➔ Device] 호스트 제어 프로토콜 (Command Channel)

### 2.1. 32바이트 Command Struct 구조

- **Struct Format:** `<4H 4i f I` (리틀 엔디안, 패딩 없음)
- **Total Size:** 32 Bytes

| Offset | 바이트 | 데이터형 | 변수명 | 설명 |
|--------|--------|---------|--------|------|
| `0~1` | 2 | `uint16` | **sync** | 고정값 `0xBB66` (PC 발신 증명 및 동기화) |
| `2~3` | 2 | `uint16` | **cmd_id** | 수행할 명령 코드 (§2.2 참조) |
| `4~5` | 2 | `uint16` | **target_id** | 명령 대상 툴의 고유 ID (전체 제어 시 `0`) |
| `6~7` | 2 | `uint16` | **target_type** | 대상 툴의 타입 코드 (§2.4 참조). 전체 제어 시 `0`. Result Struct의 `tool_type`과 대칭 |
| `8~11` | 4 | `int32` | **param[0]** | 다목적 정수형 파라미터 0 (명령별 의미 다름, §2.3 참조) |
| `12~15` | 4 | `int32` | **param[1]** | 다목적 정수형 파라미터 1 |
| `16~19` | 4 | `int32` | **param[2]** | 다목적 정수형 파라미터 2 |
| `20~23` | 4 | `int32` | **param[3]** | 다목적 정수형 파라미터 3 |
| `24~27` | 4 | `float32` | **fparam** | 부동소수점 파라미터 (기본 용도: 회전각 `rot_angle`. 명령별로 재정의 가능. 미사용 시 `0.0`) |
| `28~31` | 4 | `uint32` | **payload_len** | 헤더 직후 이어지는 추가 바이너리 데이터의 바이트 크기 (없으면 `0`) |

```python
# Python / PC 측 참조 구현
CMD_FORMAT = '<4H 4i f I'   # 32 bytes, zero padding
CMD_SYNC   = 0xBB66

def pack_command(cmd_id, target_id=0, target_type=0,
                 p0=0, p1=0, p2=0, p3=0, fparam=0.0, payload_len=0):
    return struct.pack(CMD_FORMAT,
                       CMD_SYNC, cmd_id, target_id, target_type,
                       p0, p1, p2, p3, fparam, payload_len)
```

```c
/* C / MicroPython(firmware) 측 참조 구현 */
typedef struct __attribute__((packed)) {
    uint16_t sync;          /* 0xBB66 */
    uint16_t cmd_id;
    uint16_t target_id;
    uint16_t target_type;
    int32_t  param[4];
    float    fparam;
    uint32_t payload_len;
} HyCmd_t;                  /* sizeof = 32 */
```

> **설계 메모:**
> - `sync`·`cmd_id`·`target_id`·`target_type` 모두 2바이트로 통일 → 자연 정렬(2/4바이트 경계), 컴파일러 암묵적 패딩 발생 없음.
> - `target_id` uint8→uint16 확장으로 Result Struct의 `tool_id`(uint16)와 완전 대칭.
> - `param[1]`은 v2.0에서 `tool_type`을 담았으나, `target_type`이 헤더로 승격되어 해제됨. 다목적 정수 슬롯으로 전환.
> - `fparam`은 `rot_angle`이 기본 용도이며, SET_TOOL 명령에서 Search ROI 전체(x, y, w, h, angle)를 32B 헤더만으로 표현 가능하게 한다.

### 2.2. Command ID (cmd_id) 전체 목록

#### [시스템 상태 제어] (payload_len = 0)

| ID | 상수명 | 설명 | param / fparam 사용 |
|----|--------|------|---------------------|
| `0x0000` | `STOP` | 모든 연산/송출 중지 → `STANDBY` 모드 진입 | 미사용 |
| `0x0001` | `LIVE` | 비전 연산 OFF, 최고화질 원시 영상(Raw Frame) 지속 송출 | `param[0]` = FPS 상한 (0이면 무제한) |
| `0x0002` | `TEACH_SNAP` | 현재 프레임 1장 캡처 후 송출, 이후 `STANDBY` 유지 | 미사용 |
| `0x0003` | `TEST` | 비전 연산 ON, 결과(Burst) + 압축 영상 동시 지속 송출 | `param[0]` = 영상 송출 주기 (0=매 프레임) |
| `0x0004` | `RUN` | 비전 연산 ON, 결과(Burst)만 초고속 송출 (영상 OFF) | `param[0]` = Burst-only 플래그 |

#### [레시피 DCOM 트리 프로비저닝]

| ID | 상수명 | 설명 | param / fparam 사용 |
|----|--------|------|---------------------|
| `0x0010` | `SET_TOOL` | target_id에 해당하는 비전 툴 노드를 생성/수정 | §2.3 참조 |
| `0x0011` | `SET_TEMPLATE` | 패턴 매칭용 템플릿 이미지 등 무거운 바이너리 데이터 세팅 | `param[0]` = 템플릿 인덱스, `payload_len` = N바이트 |
| `0x0012` | `SET_CAMERA` | 카메라 센서 설정 (노출, 게인, 화질 등) | `param[0~3]` = 센서 파라미터 압축 |
| `0x0018` | `SAVE_RECIPE` | 현재 RAM 상의 레시피를 장치 Flash에 영구 저장 | `param[0]` = 슬롯 번호 |
| `0x0019` | `CLEAR_TOOLS` | 장치 RAM에 등록된 모든 비전 툴 초기화 | 미사용 |

#### [진단/유틸리티]

| ID | 상수명 | 설명 | param / fparam 사용 |
|----|--------|------|---------------------|
| `0x00F0` | `PING` | 연결 상태 확인 (장치는 `0xF1 PONG`으로 응답) | 미사용 |
| `0x00FF` | `RESET` | 장치 하드웨어 소프트 리셋 | 미사용 |

### 2.3. SET_TOOL (0x0010) 파라미터 상세

`SET_TOOL`의 헤더와 `param[0~3]` + `fparam`은 **트리 위상 + Search ROI 전체 geometry**를 32B 안에 완전히 인코딩합니다.

| 필드 | 인코딩 | 설명 |
|------|--------|------|
| `target_id` | `uint16` | 생성/수정할 툴의 고유 ID |
| `target_type` | `uint16` | 툴의 종류 코드 (§2.4) |
| `param[0]` | 상위 16bit: `device_id`, 하위 16bit: `parent_tool_id` | **트리 조립 정보.** `parent_tool_id = 0`이면 루트 노드 |
| `param[1]` | 다목적 (툴 타입별 정의) | 툴 타입별 추가 정수 파라미터 (예: `HyLocator`의 `update_policy`) |
| `param[2]` | 상위 16bit: `x`, 하위 16bit: `y` | Search ROI 좌상단 좌표 (픽셀, 압축) |
| `param[3]` | 상위 16bit: `w`, 하위 16bit: `h` | Search ROI 크기 (픽셀, 압축) |
| `fparam` | `float32` | Search ROI 회전각 `rot_angle` (도(°) 단위, 미사용 시 `0.0`) |
| `payload_len` | `uint32` | 툴별 확장 파라미터(판정 임계값 등) 페이로드 크기. 기본 등록 시 `0` |

> **SET_TOOL 32B 완결성:** `target_type`이 헤더에, `rot_angle`이 `fparam`에 들어감으로써,
> Search ROI의 5개 요소 **(x, y, w, h, angle)** 가 추가 payload 없이 32B 헤더만으로 완전히 전달됩니다.
> 판정 임계값·통계 설정 등 툴별 확장 속성은 `payload`로 추가 전달합니다.

#### payload 구조체 (툴 타입별, 선택 전송)

| tool_type | payload 포맷 | 설명 |
|-----------|-------------|------|
| `TOOL_LINE` | `<f f f f` (16B) | `th_lo`, `th_hi`, `min_cnt`, `max_cnt` |
| `TOOL_LOCATOR` | `<H H f f` (12B) | `update_policy`, `reserved`, `allow_angle_min`, `allow_angle_max` |
| `TOOL_PATMAT` | `<f f H H H H` (16B) | `th_scan`, `th_find`, `step_l0`, `enable_L1`, `enable_L2`, `sig_bins` |
| `TOOL_AND` / `TOOL_OR` | — | children_ids는 별도 `ADD_REF` 명령으로 전달 |
| 기타 | 툴 추가 시 정의 | |

## 3. [Device ➔ PC] 결과 프로토콜 (Result Channel)

### 3.1. Burst 전송 흐름 (Stream Sequence)

장치는 1회 사이클(또는 바톤 터치 시점) 연산이 끝나면 아래 순서대로 버퍼를 밀어냅니다:

```
[1] Master Sync  :  0xAA55          (2 Bytes, uint16)
[2] Packet Count :  N               (2 Bytes, uint16)
[3] Result Data  :  64B × N개       (연속 전송)
[4] Image Size   :  M               (4 Bytes, uint32)
[5] Image Data   :  M Bytes JPEG    (RUN 모드 등 영상 OFF 시 M = 0)
```

### 3.2. 64바이트 Result Struct 구조 (단일 패킷)

- **Struct Format:** `<H I I I H H H B B 5f 4f I 2x` (리틀 엔디안, 공백 없음)
- **Total Size:** 64 Bytes

| Offset | 바이트 | 데이터형 | 변수명 | 설명 |
|--------|--------|---------|--------|------|
| `0~1` | 2 | `uint16` | **sync** | 고정값 `0xAA55` |
| `2~5` | 4 | `uint32` | **tx_id** | 통신 트랜잭션 시퀀스 번호 (1~65535 순환) |
| `6~9` | 4 | `uint32` | **cycle_id** | 물리적 부품 1개당 부여되는 생명주기 고유 번호 |
| `10~13` | 4 | `uint32` | **img_id** | 프레임(이미지) 캡처 고유 번호 (캐싱/중복 방지) |
| `14~15` | 2 | `uint16` | **seq_id** | 트리 내 실행 순서 |
| `16~17` | 2 | `uint16` | **tool_id** | 툴 고유 식별자 (**상태 매핑 키**: PC 측 트리에서 이 ID로 노드를 찾아 상태 갱신) |
| `18~19` | 2 | `uint16` | **tool_type** | 툴의 종류 코드 (§2.4 참조). Command의 `target_type`과 대칭 |
| `20` | 1 | `uint8` | **rst_done** | **실행 상태** (§3.3 참조) |
| `21` | 1 | `uint8` | **rst_state** | **논리 판정 결과** (§3.3 참조) |
| `22~41` | 20 | `float32 × 5` | **geometry** | `x, y, w, h, angle` (결과 좌표 및 규격) |
| `42~57` | 16 | `float32 × 4` | **stats** | `stat1~4` (Score, 대비, 편차 등 다목적 통계치) |
| `58~61` | 4 | `uint32` | **proc_time** | 해당 툴 단독 연산 소요 시간 (밀리초) |
| `62~63` | 2 | `Pad` | **padding** | 64바이트 정렬용 |

### 3.3. 실행/판정 상태 코드 (Tri-State 체계)

`rst_done`과 `rst_state`는 **3-상태(Tri-State)** 체계를 사용하여 `HyWhen`의 `Pending` 상태를 프로토콜 수준에서 완벽하게 지원합니다.

#### `rst_done` — 실행 완료 상태

| 값 | 상수명 | 의미 |
|----|--------|------|
| `0` | `EXEC_IDLE` | 미실행 (단축 평가에 의한 스킵 포함) |
| `1` | `EXEC_DONE` | 정상 완료 |
| `2` | `EXEC_ERROR` | 실행 중 에러 발생 (`usable_devices` 불일치 포함) |
| `3` | `EXEC_PENDING` | 진행 중 / 대기 (HyWhen 타이머 가동 중 등) |

#### `rst_state` — 논리 판정 결과

| 값 | 상수명 | 의미 |
|----|--------|------|
| `0` | `JUDGE_NG` | NG / False / 불합격 |
| `1` | `JUDGE_OK` | OK / True / 합격 |
| `2` | `JUDGE_PENDING` | 미확정 (HyWhen이 아직 조건을 충족하지 못함) |

> **설계 의도:** `HyAnd` 컨테이너는 자식 중 `rst_state == JUDGE_PENDING`인 노드가 있으면 단축 평가를 보류하고 전체 시퀀스를 `PENDING`으로 유지합니다. 이를 통해 런타임 중 데이터가 확정되지 않아 발생하는 섣부른 오판을 방어합니다.

## 4. 동기화 및 예외 처리 (Sync & Exception Handling)

### 4.1. Sync Header 불일치 방어

PC 또는 장치는 데이터 수신 중 동기화 헤더(`0xBB66` 또는 `0xAA55`)가 맞지 않을 경우:

1. **1바이트씩 슬라이딩 윈도우 스캔**을 수행
2. 다음 유효한 동기화 헤더를 찾을 때까지 데이터를 폐기(Drop)
3. 이를 통해 블랙홀(무한 대기) 현상을 완벽 방어

### 4.2. OOM (Out Of Memory) 방어

- Payload 수신 전 `target_type` 필드로 예상 버퍼 크기를 사전 계산 가능 (타입별 payload 크기가 알려져 있으므로).
- 할당할 수 있는 최대 버퍼 크기는 장치의 RAM 한계를 고려하여 **하드코딩된 임계값**을 초과할 수 없습니다.
- 초과 시 해당 트랜잭션을 무시하고 입력 버퍼를 비웁니다.

### 4.3. target_type 불일치 방어

수신된 `target_id`로 레지스트리에서 툴을 조회했을 때 등록된 타입과 `target_type`이 다른 경우:

1. 해당 명령을 무시하고 경고 로그 기록
2. `rst_done = EXEC_ERROR`로 응답 (가능한 경우)

### 4.4. 연결 상태 감시 (Heartbeat)

- PC는 주기적으로 `PING (0x00F0)` 명령을 전송합니다.
- 장치는 수신 즉시 **Result Struct 1개** (`tool_id = 0`, `rst_done = EXEC_DONE`)를 Burst 형태로 응답합니다.
- 3회 연속 Timeout(각 1초) 시 PC는 연결 단절로 판정하고 UI 상태를 `DISCONNECTED`로 전환합니다.

### 4.5. Flow Control (Back-Pressure)

장치가 Cycle N의 Burst를 전송한 직후 Cycle N+1의 캡처를 시작하는 파이프라인 구조에서:

- PC 측 수신 큐의 **최대 깊이는 3 사이클**로 제한합니다.
- 큐가 가득 찬 경우, 가장 오래된 미처리 사이클을 폐기(Drop)하고 경고 로그를 남깁니다.
- 향후 장치 측에 `FLOW_PAUSE / FLOW_RESUME` 명령을 추가하여 명시적 흐름 제어를 지원할 수 있습니다.

## 5. Command / Result 필드 대칭 요약

Command(PC→Device)와 Result(Device→PC) 간 대응 필드를 명시합니다.
