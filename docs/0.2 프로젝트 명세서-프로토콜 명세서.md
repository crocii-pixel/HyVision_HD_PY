# 프로토콜 명세서 (Protocol Specification)

**버전:** 2.0 (마스터 상세서 v4.3 동기화)
**최종 수정:** 2026-04-14

## 1. 설계 원칙

### 1.1. 고정 길이 이진 구조체 (Fixed-Length Binary Struct)

가변 길이 문자열(JSON 등) 파싱으로 인한 마이크로컨트롤러의 메모리 파편화 및 병목 현상을 원천 차단하기 위해, 모든 런타임 통신에 **고정 길이 이진 구조체**를 사용합니다.

### 1.2. 바이트 오더 (Byte Order)

- **모든 구조체는 리틀 엔디안(Little-Endian, `<`)** 으로 인코딩/디코딩합니다.
- PC(x86/x64)와 ARM Cortex-M7(OpenMV H7 Plus) 모두 리틀 엔디안 네이티브이므로, 바이트 스왑 없이 직접 캐스팅 가능합니다.

### 1.3. 통신 방향 및 채널 크기

| 방향 | 채널명 | 단위 크기 | 설명 |
|------|--------|----------|------|
| **PC ➔ Device** | 제어 채널 (Command Channel) | `32 Bytes` 고정 헤더 + 선택적 바이너리 페이로드 | 모드 전환, 레시피(트리) 주입 |
| **Device ➔ PC** | 결과 채널 (Result Channel) | `64 Bytes` × N개 배열(Burst) + 영상 페이로드 | 트리 상태 동기화, 이미지 전송 |

## 2. [PC ➔ Device] 호스트 제어 프로토콜 (Command Channel)

### 2.1. 32바이트 Command Struct 구조

- **Struct Format:** `<H B B 4i I 8x` (리틀 엔디안)
- **Total Size:** 32 Bytes

| Offset | 바이트 | 데이터형 | 변수명 | 설명 |
|--------|--------|---------|--------|------|
| `0~1` | 2 | `uint16` | **sync** | 고정값 `0xBB66` (PC 발신 증명 및 동기화) |
| `2` | 1 | `uint8` | **cmd_id** | 수행할 명령 코드 (§2.2 참조) |
| `3` | 1 | `uint8` | **target_id** | 명령 대상 툴의 고유 ID (전체 제어 시 `0`) |
| `4~7` | 4 | `int32` | **param[0]** | 다목적 정수형 파라미터 1 (명령별 의미 다름, §2.3 참조) |
| `8~11` | 4 | `int32` | **param[1]** | 다목적 정수형 파라미터 2 |
| `12~15` | 4 | `int32` | **param[2]** | 다목적 정수형 파라미터 3 |
| `16~19` | 4 | `int32` | **param[3]** | 다목적 정수형 파라미터 4 |
| `20~23` | 4 | `uint32` | **payload_len** | 헤더 직후 이어지는 추가 바이너리 데이터의 바이트 크기 (없으면 `0`) |
| `24~31` | 8 | `Pad` | **reserved** | 32바이트 정렬 및 향후 확장용 예약 여백 |

### 2.2. Command ID (cmd_id) 전체 목록

#### [시스템 상태 제어] (payload_len = 0)

| ID | 상수명 | 설명 | param 사용 |
|----|--------|------|-----------|
| `0x00` | `STOP` | 모든 연산/송출 중지 → `STANDBY` 모드 진입 | 미사용 |
| `0x01` | `LIVE` | 비전 연산 OFF, 최고화질 원시 영상(Raw Frame) 지속 송출 | `param[0]` = FPS 상한 (0이면 무제한) |
| `0x02` | `TEACH_SNAP` | 현재 프레임 1장 캡처 후 송출, 이후 `STANDBY` 유지 | 미사용 |
| `0x03` | `TEST` | 비전 연산 ON, 결과(Burst) + 압축 영상 동시 지속 송출 | `param[0]` = 영상 송출 주기 (0=매 프레임) |
| `0x04` | `RUN` | 비전 연산 ON, 결과(Burst)만 초고속 송출 (영상 OFF) | `param[0]` = Burst-only 플래그 (0=결과만, 1=선택적 영상 포함) |

#### [레시피 DCOM 트리 프로비저닝] (트리 노드 생성/수정/삭제)

| ID | 상수명 | 설명 | param 사용 |
|----|--------|------|-----------|
| `0x10` | `SET_TOOL` | target_id에 해당하는 비전 툴 노드를 생성/수정 (payload = 0) | §2.3 참조 |
| `0x11` | `SET_TEMPLATE` | 패턴 매칭용 템플릿 이미지 등 무거운 바이너리 데이터 세팅 | `param[0]` = 템플릿 인덱스, `payload_len` = N바이트 |
| `0x12` | `SET_CAMERA` | 카메라 센서 설정 (노출, 게인, 화질 등) | `param[0~3]` = 센서 파라미터 압축 |
| `0x18` | `SAVE_RECIPE` | 현재 RAM 상의 레시피를 장치 Flash에 영구 저장 | `param[0]` = 슬롯 번호 |
| `0x19` | `CLEAR_TOOLS` | 장치 RAM에 등록된 모든 비전 툴 초기화 | 미사용 |

#### [진단/유틸리티]

| ID | 상수명 | 설명 | param 사용 |
|----|--------|------|-----------|
| `0xF0` | `PING` | 연결 상태 확인 (장치는 `0xF1 PONG`으로 응답) | 미사용 |
| `0xFF` | `RESET` | 장치 하드웨어 소프트 리셋 | 미사용 |

### 2.3. SET_TOOL (0x10) 파라미터 상세

`SET_TOOL` 명령의 `param[0~3]`은 트리의 구조(위상)와 노드의 로컬 속성을 동시에 인코딩합니다.

| 파라미터 | 인코딩 | 설명 |
|----------|--------|------|
| `param[0]` | 상위 16bit: `device_id`, 하위 16bit: `parent_tool_id` | **트리 조립 정보.** `parent_tool_id = 0`이면 루트 노드. PC가 DFS 순서로 순차 송출하면 장치가 트리를 재조립 가능 |
| `param[1]` | `tool_type` (§2.4 참조) | 툴의 종류 코드 |
| `param[2]` | 상위 16bit: `x`, 하위 16bit: `y` | Search ROI 좌상단 좌표 (압축) |
| `param[3]` | 상위 16bit: `w`, 하위 16bit: `h` | Search ROI 크기 (압축) |

> **참고:** 이전 프로토콜 명세서(v1.x)와의 차이 — `param[0]`에 `tool_type`이 아닌 **트리 부모-자식 관계**가 들어갑니다. 이는 마스터 상세서 v4.3의 DCOM Tree Serialization 정책을 반영한 것입니다.

### 2.4. Tool Type 코드 (전체 Enum)

| 코드 | 상수명 | 분류 | 주 할당 장치 |
|------|--------|------|-------------|
| `0x01` | `TOOL_LINE` | 물리 비전 | Device (1) |
| `0x02` | `TOOL_PATMAT` | 물리 비전 | Device (1) |
| `0x03` | `TOOL_LOCATOR` | 물리 비전 | Device (1) |
| `0x04` | `TOOL_INTERSECTION` | 물리 비전 | Device (1) |
| `0x05` | `TOOL_LINE_PATMAT` | 물리 비전 | Device (1) |
| `0x10` | `TOOL_DISTANCE` | 정밀 측정 | PC (2) |
| `0x11` | `TOOL_CONTRAST` | 정밀 측정 | PC (2) |
| `0x12` | `TOOL_FND` | 정밀 측정 | PC (2) |
| `0x20` | `TOOL_WHEN` | 로직 집행관 | PC (2) |

| `0x22` | `TOOL_OR` | 로직 집행관 | PC (2) |
| `0x23` | `TOOL_FIN` | 로직 집행관 | PC (2) |

## 3. [Device ➔ PC] 결과 프로토콜 (Result Channel)

### 3.1. Burst 전송 흐름 (Stream Sequence)

장치는 1회 사이클(또는 바톤 터치 시점) 연산이 끝나면 아래 순서대로 버퍼를 밀어냅니다:

```
[1] Master Sync  :  0xAA55          (2 Bytes, uint16)
[2] Packet Count :  N               (2 Bytes, uint16)
[3] Result Data  :  64B × N개       (연속 전송)
[4] Image Size   :  M               (4 Bytes, uint32)
[5] Image Data   :  M Bytes JPEG    (RUN 모드 등 영상 OFF 시 M = 0)
```

### 3.2. 64바이트 Result Struct 구조 (단일 패킷)

- **Struct Format:** `<H I I I H H H B B 5f 4f I 2x` (리틀 엔디안, 공백 없음)
- **Total Size:** 64 Bytes

| Offset | 바이트 | 데이터형 | 변수명 | 설명 |
|--------|--------|---------|--------|------|
| `0~1` | 2 | `uint16` | **sync** | 고정값 `0xAA55` |
| `2~5` | 4 | `uint32` | **tx_id** | 통신 트랜잭션 시퀀스 번호 (1~65535 순환) |
| `6~9` | 4 | `uint32` | **cycle_id** | 물리적 부품 1개당 부여되는 생명주기 고유 번호 |
| `10~13` | 4 | `uint32` | **img_id** | 프레임(이미지) 캡처 고유 번호 (캐싱/중복 방지) |
| `14~15` | 2 | `uint16` | **seq_id** | 트리 내 실행 순서 |
| `16~17` | 2 | `uint16` | **tool_id** | 툴 고유 식별자 (**상태 매핑 키**: PC 측 트리에서 이 ID로 노드를 찾아 상태 갱신) |
| `18~19` | 2 | `uint16` | **tool_type** | 툴의 종류 코드 (§2.4 참조) |
| `20` | 1 | `uint8` | **rst_done** | **실행 상태** (아래 §3.3 참조) |
| `21` | 1 | `uint8` | **rst_state** | **논리 판정 결과** (아래 §3.3 참조) |
| `22~41` | 20 | `float32 × 5` | **geometry** | `x, y, w, h, angle` (결과 좌표 및 규격) |
| `42~57` | 16 | `float32 × 4` | **stats** | `stat1~4` (Score, 대비, 편차 등 다목적 통계치) |
| `58~61` | 4 | `uint32` | **proc_time** | 해당 툴 단독 연산 소요 시간 (밀리초) |
| `62~63` | 2 | `Pad` | **padding** | 64바이트 정렬용 |

### 3.3. 실행/판정 상태 코드 (Tri-State 체계)

`rst_done`과 `rst_state`는 **3-상태(Tri-State)** 체계를 사용하여 `HyWhen`의 `Pending` 상태를 프로토콜 수준에서 완벽하게 지원합니다.

#### `rst_done` — 실행 완료 상태

| 값 | 상수명 | 의미 |
|----|--------|------|
| `0` | `EXEC_IDLE` | 미실행 (단축 평가에 의한 스킵 포함) |
| `1` | `EXEC_DONE` | 정상 완료 |
| `2` | `EXEC_ERROR` | 실행 중 에러 발생 |
| `3` | `EXEC_PENDING` | 진행 중 / 대기 (HyWhen 타이머 가동 중 등) |

#### `rst_state` — 논리 판정 결과

| 값 | 상수명 | 의미 |
|----|--------|------|
| `0` | `JUDGE_NG` | NG / False / 불합격 |
| `1` | `JUDGE_OK` | OK / True / 합격 |
| `2` | `JUDGE_PENDING` | 미확정 (HyWhen이 아직 조건을 충족하지 못함) |

> **설계 의도:** `HyAnd` 컨테이너는 자식 중 `rst_state == JUDGE_PENDING`인 노드가 있으면 단축 평가를 보류하고 전체 시퀀스를 `PENDING`으로 유지합니다. 이를 통해 런타임 중 데이터가 확정되지 않아 발생하는 섣부른 오판을 방어합니다.

## 4. 동기화 및 예외 처리 (Sync & Exception Handling)

### 4.1. Sync Header 불일치 방어

PC 또는 장치는 데이터 수신 중 동기화 헤더(`0xBB66` 또는 `0xAA55`)가 맞지 않을 경우:

1. **1바이트씩 슬라이딩 윈도우 스캔**을 수행
2. 다음 유효한 동기화 헤더를 찾을 때까지 데이터를 폐기(Drop)
3. 이를 통해 블랙홀(무한 대기) 현상을 완벽 방어

### 4.2. OOM (Out Of Memory) 방어

- Payload 데이터 수신 전 할당할 수 있는 최대 버퍼 크기는 장치의 RAM 한계를 고려하여 **하드코딩된 임계값**을 초과할 수 없습니다.
- 초과 시 해당 트랜잭션을 무시하고 입력 버퍼를 비웁니다.

### 4.3. 연결 상태 감시 (Heartbeat)

- PC는 주기적으로 `PING (0xF0)` 명령을 전송합니다.
- 장치는 수신 즉시 **Result Struct 1개** (`tool_id = 0`, `rst_done = EXEC_DONE`)를 Burst 형태로 응답합니다.
- 3회 연속 Timeout(각 1초) 시 PC는 연결 단절로 판정하고 UI 상태를 `DISCONNECTED`로 전환합니다.

### 4.4. Flow Control (Back-Pressure)

장치가 Cycle N의 Burst를 전송한 직후 Cycle N+1의 캡처를 시작하는 파이프라인 구조에서:

- PC 측 수신 큐의 **최대 깊이는 3 사이클**로 제한합니다.
- 큐가 가득 찬 경우, 가장 오래된 미처리 사이클을 폐기(Drop)하고 경고 로그를 남깁니다.
- 향후 장치 측에 `FLOW_PAUSE / FLOW_RESUME` 명령을 추가하여 명시적 흐름 제어를 지원할 수 있습니다.
